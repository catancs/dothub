The idea

The whole environment, shared and discoverable

Not another skills directory. The bundle is your entire agent setup, and the discovery layer is social.

People pour real effort into their Claude Code harness — CLAUDE.md, skills, custom commands, subagents, hooks, MCP servers. Today that work lives in private dotfile repos, or gets shared one fragment at a time on component directories. dothub hosts the whole bundle, versioned, behind a public feed — so you can publish yours, see what smart people actually run, and pull a setup into your own machine in one move.

Why this is open territory

Research (2026-06-29) found the pieces exist but split along a fracture nobody bridges: component registries (Smithery / cursor.directory / Open Agent Skills) are social but single-artifact; sync tools (ai-dotfiles-manager / Microsoft APM / MCP Market Hub) are whole-environment but private/team-only. The missing tile is whole-environment bundles + a public social feed + agent-native push.

Flows

Walk through it

Four flows. Watch who does what — you agent server db. Note the agent does all local filesystem work; the server is a pure registry.

A Publish — agent self-push
  1. you “Publish my setup as my-python-flow.”
  2. agent reads its own ~/.claude/ — skills, commands, agents, hooks/, .mcp.json, CLAUDE.md — using its file tools.
  3. agent builds a {path: content} map and calls publish_setup(title, description, files) on the remote MCP server (auth: your API key).
  4. server validates (path-traversal guard, size cap, secret scan), tars the entries, parses the manifest.
  5. db inserts a setup_version with version = latest + 1.
  6. server returns {slug, version, url} → your setup is live on the feed.
B Discover & install — pull someone’s setup
  1. you browse the feed, or ask “list popular setups.”
  2. agent calls list_setups() → server returns ranked cards (with a ⚠ on any that run code).
  3. you “Install some-user/claude-setup.”
  4. agent calls preview_setup(slug) → server returns the effects manifest (exact hook commands, MCP servers) + file list.
  5. agent shows you a diff vs your current .claude/, the exact commands, and what runs code — then waits.
  6. you approve. Only now agent calls install_setup, db bumps downloads, files are written, /reload.
C Browse — the human web feed
  1. you open / → a feed of public setups (title, author, downloads, component summary).
  2. you open /s/{slug} → detail: description, version history, file tree, and copy-paste pull instructions.
D Onboard — get a key
  1. you POST /api/signup {username, email} → {api_key} (shown once).
  2. you paste the key into your Claude Code MCP server config — done, you can now publish.

Architecture

One process, one artifact

A single FastAPI app is both the remote MCP server and the web feed. Same binary on AWS and the VPS — only env vars differ.

Claude Code agent + browser
└▶ Nginx — TLS, reverse proxy
   └▶ gunicorn / uvicorn (systemd) → FastAPI
      ├─ GET /      HTML feed
      ├─ GET /s/{slug} detail
      ├─ /api/...   JSON REST
      └─ /mcp      remote MCP (FastMCP)
   └▶ Postgres — users · setups · setup_versions

The boundary that keeps it simple: the server is only a registry. The agent (local) does every filesystem read/write. The backend never touches your disk, so it stays a clean stateless networked service.

Surface

API & MCP tools

Both call the same underlying logic. REST for humans/curl; MCP tools for the agent.

REST (browser · curl)

  • POST /api/signup · POST /api/login
  • POST /api/keys → api_key · DELETE revoke
  • GET /api/setups — public list
  • GET /api/setups/{slug} — + versions + effects
  • POST /api/setups (auth) — create / bump
  • GET /api/setups/{slug}/download — presigned S3 URL

MCP tools (agent-native)

  • publish_setup(title, description, files, slug?)
  • preview_setup(slug) → effects + file list
  • install_setup(slug) → files + effects
  • list_setups(query?) → incl. runs_code flag

Publish semantics: new slug → create (owned by you) · your existing slug → bump version · someone else’s slug → 403.

Deployment — the actual point

One robust AWS stack

One robust AWS cloud-native stack — the most architecture-rich target. (CI/CD done elsewhere; bare-metal VPS skipped.)

Compute · edge

EC2 + Nginx

EC2 (Ubuntu) running gunicorn + uvicorn under systemd; Nginx reverse proxy + certbot TLS on the instance.

Data

RDS + S3

Managed RDS Postgres for metadata; S3 for bundle bytes, reached via the EC2 IAM role — no static keys on the box.

Networking

Security groups

80/443 open · 22 from your IP only · RDS 5432 only from the EC2 group · S3 scoped by IAM. The real architecture, wired for real.

What’s missing? Pressure-test it.

The honest gaps. Some are now planned fast-follows (§11 in the spec); the rest are still open for your input.

✓ Decided: transparency-first install, responsibility on the user

A setup bundle includes hooks/ (shell commands) and .mcp.json (server processes), so installing one is effectively running someone’s code. v1’s answer is maximum visibility, not sandboxing: the server computes an effects manifest at publish (every hook + its exact command, every MCP server, a runs_code flag); on install the agent shows a diff vs your current .claude/ and those exact commands, and waits for your explicit approval.

The platform makes effects as visible as possible; verifying safety is the user’s responsibility. If abuse appears later, the next lever is verified/signed authors — not server-side sandboxing.
