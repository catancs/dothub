Design comparison

Tag density on the feed card

Each setup auto-derives tags from its effects manifest: hooks, mcp:linear, plugins, commands, skills-only. The question is how many of those show on a feed card before it gets noisy. Full tag list always appears on the detail page either way.

Option A recommended
Primary tag only. One pill: the single most descriptive signal (skills-only for no-code setups, hooks for code-running ones). Clean, scannable.

Option B
All tags. Every derived tag shows as a pill. More information per card, but a setup with hooks + MCP + plugins + commands can show four or five pills alongside the runs-code chip.

How the primary tag is chosen (Option A): prefer skills-only when runs_code is false (it is the safest, most decision-relevant signal). Otherwise prefer hooks (code runs), then mcp:<name>, then plugins, then the first non-zero count tag. The full tag set still appears on the detail page header in both options, so no information is lost, only card noise is reduced.
