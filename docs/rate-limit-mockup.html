Design comparison

How to rate-limit auth

Login, signup, and API-key minting are unbounded today. A bot on a t3.micro brute-forces the login page in minutes. Both options below stop that. The question is whether to add a dependency or hand-roll ~30 lines.

Option A recommended
slowapi, the standard FastAPI rate limiter. Decorator on each route.
New dependencyyes, slowapi
Lines I write~10 (decorators + setup)
Edge caseshandled (X-Forwarded-For, 429 responses, per-route limits)
Maintenancelibrary owns it
In-memory only?yes (resets on restart, fine for single-instance)
Costone pip install
# main.py
from slowapi import Limiter
limiter = Limiter(key_func=lambda r: r.client.host)
app.state.limiter = limiter

# auth.py
@router.post("/login")
@limiter.limit("10/minute")
def login_submit(request, identifier=Form(...), ...):

Trade-offs

  • Battle-tested, handles forwarded IPs behind nginx
  • Per-route limits via a readable decorator
  • Returns proper 429 with Retry-After
  • One new dependency to pin and update
Option B
Hand-rolled in-memory limiter. A dict of IP to recent timestamps, pruned each request.
New dependencynone, zero-dep
Lines I write~30 (the limiter + wiring)
Edge casesI handle X-Forwarded-For, 429, pruning myself
MaintenanceI own it forever
In-memory only?yes (same caveat, single-instance)
Costmy time + future bug surface
# app/ratelimit.py
_hits: dict[str, list[float]] = {}
def check(ip: str, limit: int, window: float):
    now = monotonic()
    hits = [t for t in _hits.get(ip, [])
            if t > now - window]
    if len(hits) >= limit:
        raise HTTPException(429)
    hits.append(now); _hits[ip] = hits

# then call check() at the top of each route,
# reading X-Forwarded-For behind nginx yourself

Trade-offs

  • No new dependency, nothing to pin or update
  • Full control over the logic
  • I reimplement what slowapi already does correctly
  • X-Forwarded-For parsing behind nginx is easy to get wrong
  • Future me maintains a bespoke limiter
Both are in-memory, which means the counter resets if the EC2 restarts. That is fine for dothub's single-instance deploy: a restart is rare and a brief window of unlimited attempts is not a real exposure. If this ever scales to multiple instances, the limiter would move to Redis, but that is out of scope for a learning-project launch. The real difference is whether I spend one dependency or thirty lines of code I then own.
